【會議逐字稿範例】
日期：2026年4月11日
主題：Q2 產品路線圖規劃會議

---

王小明（產品經理）：好，大家都到齊了，我們開始。今天主要討論 Q2 的產品優先級，以及幾個技術債的處理方式。首先，我先說一下目前的狀況，上個月我們的用戶留存率掉了大概 8%，主要原因是載入速度太慢，用戶在第一次開啟 App 的時候，等待超過 5 秒就離開了。

李小華（工程師）：對，我看了一下 APM 的資料，問題主要集中在首頁的 API，有個 N+1 query 的問題一直沒有被修掉，每次首頁載入會打大概 40~50 次 DB query。這個其實不難修，我估計兩天內可以解決。

陳大偉（工程師）：我補充一下，除了 N+1 的問題，我們的圖片也沒有做 CDN，所有圖片都是直接從 origin server 拉，這也是一個大問題。特別是東南亞的用戶，延遲非常高。

王小明：所以這兩件事要先處理？

李小華：對，我建議這週先把 N+1 修掉，下週評估 CDN 方案。CloudFront 或 Cloudflare 都可以，我有做過一些比較，Cloudflare 的費用比較低，而且設定比較簡單。

林美珍（設計師）：我這邊也有一個問題，我們的 onboarding 流程被很多用戶反應太複雜，有個調查說新用戶完成 onboarding 的比例只有 40%，代表 60% 的新用戶根本還沒開始用就放棄了。

王小明：這個數字很嚴重。你有沒有想法要怎麼改？

林美珍：我有做了一個新的設計稿，把步驟從 7 步縮短到 3 步，把非必要的欄位都移到設定頁。下週我可以分享給大家看。

陳大偉：技術上來說應該可以配合，但如果改 onboarding 的話要注意 A/B test 的設計，不能直接全量上線。

王小明：同意。那這樣我們先做個決定：效能優化是最高優先級，李小華這週修 N+1，下週我們開一個 CDN 評估會議。林美珍下週分享新的 onboarding 設計，之後安排 A/B test。

李小華：OK，我這週 commit 要修 N+1 的問題，順便把慢查詢的 index 也補一下。

陳大偉：我來負責研究 CDN 方案，準備比較表。

林美珍：我來完成新的 onboarding 設計稿，下週三分享。

王小明：好，還有一個事情，行銷那邊希望我們 Q2 要出一個「團隊協作」的功能，讓多人可以共用帳號底下的資料。這個功能的需求大嗎？

李小華：技術複雜度很高。要做角色權限（RBAC）、資料隔離、通知系統……這個至少要一個月。

王小明：那 Q2 可能放不進去。我待會去跟行銷同步一下，可能要往 Q3 移。

陳大偉：如果要做這個功能，建議先架設好 RBAC 的基礎，避免之後重工。

王小明：好，這個先列為技術前置工作，不排入 Q2 交付，但可以開始設計架構。

王小明：好，今天就到這裡，我來整理一下會議記錄，等等發到 Slack channel。散會！
